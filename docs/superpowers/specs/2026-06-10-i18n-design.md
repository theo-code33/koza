# kōza — Internationalisation (i18n) — Design

> Spec validée le 2026-06-10. Branche : `feat/i18n`.

## Objectif

Câbler une i18n complète FR/EN sur l'app (App Router, majorité de Server Components), extraire **toutes** les chaînes visibles en dur vers des catalogues, et offrir un sélecteur de langue persistant. FR par défaut, EN secondaire. Devise **toujours EUR** (pas de conversion). Une seule PR, couverture 100 % (fr.json et en.json complets).

## Décisions d'architecture

### Mode : next-intl 4 **sans routing i18n** (cookie/DB)

La locale n'est **pas** une dimension d'URL (`/fr/…`). Justification : app mono-utilisateur, pas d'auth, pas de SEO, pas de partage de liens localisés, EUR figé. Le CLAUDE.md impose « langue stockée en base + cookie fallback ». Le mode « sans routing » est officiellement supporté par next-intl — routes inchangées, pas de middleware de redirection, pas de restructuration sous `app/[locale]/`.

### Résolution de la locale active (par requête)

`getRequestConfig` résout dans cet ordre :

1. **Cookie `NEXT_LOCALE`** (lu via `next/headers`) s'il est valide (`fr`|`en`).
2. Sinon **DB `UserSettings.locale`**.
3. Sinon défaut **`fr`**.

Rationale : `getRequestConfig` s'exécute tôt dans le pipeline ; lire un cookie y est synchrone et sans dépendance. La DB reste la **mémoire durable** (survit à la purge du cookie) ; on ré-amorce le cookie depuis la DB quand il manque.

### Application du changement de langue

`setLocaleAction(locale)` (server action) :

1. valide la locale,
2. `prisma.userSettings.update`/upsert de la ligne `default` → `locale`,
3. `cookies().set("NEXT_LOCALE", locale)`,
4. le client appelle `router.refresh()` → rendu serveur complet dans la nouvelle langue.

## Composants & fichiers

### Nouveaux

- `src/i18n/locale.ts` — `LOCALES = ["fr", "en"] as const`, `DEFAULT_LOCALE = "fr"`, type `Locale`, `isLocale(x): x is Locale`, `resolveLocale()` (lit cookie → DB → défaut). **Testable unitairement** (prisma + cookies mockés).
- `src/i18n/request.ts` — `getRequestConfig` : `locale = await resolveLocale()`, charge `src/locales/${locale}.json`, retourne `{ locale, messages }`.
- `src/locales/fr.json`, `src/locales/en.json` — catalogues complets.
- `src/app/actions/locale.ts` — `"use server"` ; `setLocaleAction(locale)`.
- `src/components/settings/locale-toggle.tsx` — Client, `Segmented` FR/EN, calqué sur `theme-toggle.tsx` ; appelle `setLocaleAction` puis `router.refresh()`.
- `src/test/render-with-intl.tsx` (helper de test) — `renderWithIntl(ui)` enveloppe dans `NextIntlClientProvider` avec le vrai `fr.json`.

### Modifiés

- `next.config.ts` — `createNextIntlPlugin("./src/i18n/request.ts")` autour de la config existante (`output: "standalone"` conservé).
- `src/app/layout.tsx` — devient Server Component async : `await getLocale()` + `getMessages()`, `<html lang={locale}>` dynamique, `<NextIntlClientProvider messages={messages}>` autour des enfants ; `metadata` via `getTranslations` (`generateMetadata`).
- `src/app/(main)/settings/page.tsx` — remplace la ligne statique « Langue » par `<LocaleToggle />`.
- `src/lib/categories.ts`, `src/lib/subcategories.ts` — les **libellés d'affichage** quittent ces fichiers (déplacés dans les catalogues sous `categories.*` / `subcategories.*`) ; clés, couleurs, ordre **restent**. Les composants rendent via `t(...)`.
- **Tous les `.tsx` visibles (~55)** + strings user-facing dans `lib/` (mapping d'erreurs) — strings remplacées par `t(...)` (`useTranslations` côté Client, `getTranslations` côté Server).
- Tests composants existants — wrappés via `renderWithIntl` (assertions FR conservées car `fr.json` porte les mêmes textes).

## Catalogues

- Emplacement : `src/locales/{fr,en}.json` (conforme CLAUDE.md).
- **Namespaces par domaine** : `common`, `nav`, `onboarding`, `dashboard`, `incomes`, `expenses`, `budgets`, `recurring`, `settings`, `categories`, `subcategories`, `errors`.
- Format **ICU** pour interpolation/pluriels (montants injectés, compteurs, « il reste {amount} »).
- `fr.json` = source de vérité ; `en.json` = mêmes clés, traduit intégralement.

## Formatters

`formatEUR` / `formatDate` / `formatMonth` acceptent déjà `locale: "fr" | "en"`. On les **alimente avec la locale active** : `getLocale()` (Server), `useLocale()` (Client). EUR figé ; seuls les formats varient :

- montants : `1 234,56 €` (FR) / `€1,234.56` (EN) ;
- dates : `JJ/MM/AAAA` (FR) / `MM/DD/YYYY` (EN).

## Données utilisateur (non traduites)

Les contenus saisis par l'utilisateur restent tels quels : `Income.source`, `Expense.description`, `Budget.name`, `RecurringExpense.label`. Seuls les **libellés applicatifs prédéfinis** (catégories, sous-catégories, UI) sont traduits.

## Tests

- **Unitaires** : `resolveLocale` (précédence cookie → DB → défaut, locale invalide ignorée) ; `setLocaleAction` (update DB + set cookie) ; **parité des clés** `fr.json` ≡ `en.json` (même ensemble de clés, récursif).
- **Composants** : via `renderWithIntl` ; assertions FR existantes conservées.
- **Formatters** : tests fr/en déjà présents — étendus si besoin.
- Couverture lib/hooks ≥ 80 % maintenue.

## Cas limites & erreurs

- Cookie absent/invalide → fallback DB → `fr`.
- Ligne `UserSettings` absente → `setLocaleAction` upsert la ligne `default`.
- Clé de traduction manquante → attrapée par le test de parité (CI rouge).
- Onboarding ne choisit pas la langue (hors scope CLAUDE.md) → défaut `fr`.

## Hors scope

- Choix de langue pendant l'onboarding.
- Locale dans l'URL / SEO multilingue.
- Conversion de devise (EUR figé).
- Négociation `Accept-Language` (optionnelle, non retenue : défaut `fr` figé suffit).
- Notifications in-app (sous-projet séparé, en attente).

## Convention de commits

Branche `feat/i18n`. Conventional Commits, anglais, scope `i18n` (et `db`/`settings`/`lib` selon le fichier). Pas de trailer `Co-authored-by`. Commits fins et atomiques (config → catalogues → resolution → provider/layout → toggle → extraction par domaine → tests). `npm run format` + `npm run lint` + `npm run test` verts avant chaque commit ; `npm run build` (avec `DATABASE_URL` factice) pour le typecheck.
